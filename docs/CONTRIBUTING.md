# 🧰 Contribution Guidelines

> This document provides useful tips for contributing to this project.

Before submitting pull-requests, please ensure you have read through this
document. These guidelines will allow us to maintain our high standards and
will help ensure your contributions are merged without issue.

## 🔭 Application Structure

The following provides a high-level overview of this projects structure.

```text
.vscode                                   # Text editor settings.
app/                                      # Application root.
    api/                                  # API endpoint routers.
    common/                               # Global application utilities.
    server/                               # Application server and utilities.
dev/                                      # Development dependencies and tools.
docs/                                     # Additional documentation.
scripts/                                  # Project related tooling.
tests/                                    # Testing root.
    integration/                          # Integration test suite.
    unit/                                 # Unit test suite.
```

## 🪄 Making Changes

When writing your changes, please ensure you follow the [PEP8] and [PyGuide].
For changes such as shellscript contributions, please follow the respective
[Styleguide].

[pep8]: https://www.python.org/dev/peps/pep-0008/
[pyguide]: https://google.github.io/styleguide/pyguide.html
[styleguide]: https://google.github.io/styleguide/

- Create a topic branch from which to base your work.
  - This is usually the `develop` branch.
  - Check with the engineering team before targeting `release` branches.
  - Please **avoid** working directly on the `develop` branch.
- Make commits of atomic and logical units of work.
- Check your work regularly using `make run-hooks`.
- Check for unnecessary whitespace using `git diff --check` before committing.
- Make sure your commit messages are in the correct format. If the commit has
  a corresponding ticket, please reference it. See below for details.
- Ensure the necessary tests exist for the change your're submitting.

> See the [Branching Model Guide] for details on our branching and release
> strategies.

[branching model guide]: https://nvie.com/posts/a-successful-git-branching-model/

## 📜 Commit Messages

We use [commitlint] for linting commit messages. See [.commitlintrc.js] for the
reference configuration.

[commitlint]: https://github.com/conventional-changelog/commitlint/
[.commitlintrc.js]: /.commitlintrc.js

Below is an example commit message, note the location of ticket references.
Sticking to this commit message structure promotes a consistent `git log` and
makes many `git` operations easier (such as `git bisect`).

```text
[ABC-123] feat: short (72 chars or less) summary

More detailed explanatory text. Wrap it to 72 characters. The blank
line separating the summary from the body is critical (unless you omit
the body entirely).

Write your commit message in the imperative: "fix bug" and not "fixed
bug" or "fixes bug". This convention matches up with commit messages
generated by commands like git merge and git revert.

Further paragraphs come after blank lines.

- Bullet points are okay, too.
- Typically a hyphen or asterisk is used for the bullet, followed by a
  single space. Use a hanging indent.

Supporting documentation:

- https://example.com/reference-specification

Fix ABC-456, ABC-789
```
